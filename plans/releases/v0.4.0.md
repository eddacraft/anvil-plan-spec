## Release Theme

**Crosscutting Coordination & Binary-First Distribution** — v0.3 turned APS
into a workflow tool; v0.4 makes it a _coordinating_ one and ships it like a
real product. The headline is the **conductor module type**: a first-class way
to model concerns that span vertical modules (releases, audits, budgets,
migrations), with `Type: Conductor`, a template, lint awareness, and a worked
example. Around it, APS becomes a global binary you install from crates.io,
Scoop, or `cargo binstall` rather than a vendored script; gains a release-
planning addon and a compound-engineering archive; exposes its CLI to agents
over MCP; and hardens its own self-hosting with `aps audit`, new hygiene lint
rules, and a reconciled status vocabulary.

## What Ships

### Conductor / Crosscutting module type (COND)

The defining new capability. A module type for horizontal concerns that
coordinate work _across_ vertical modules.

| Area              | Detail                                                                          |
| ----------------- | ------------------------------------------------------------------------------- |
| Type marker       | `Type: Conductor` column in the metadata table (parser reads ID first, Type by name) |
| Template          | `templates/conductor.template.md` — Coordinated Modules, Cross-Module Work Items, Status Roll-up |
| Lint awareness    | **W002** (unresolved cross-module refs in a conductor) + **W006** (index entries under a Conductor section must be `Type: Conductor`) |
| Index treatment   | Optional `### Conductor / Crosscutting` index subsection (template + dogfooded) |
| Docs              | `docs/conductor-modules.md` + an `aps-rules.md` "Module Types" section          |

**APS module:** [conductor](../modules/conductor.aps.md) (COND-001…006 Complete)

### Release planning & compound engineering (REL, COMPOUND)

APS learns to narrate its own releases and compound knowledge across them.

| Area                | Detail                                                                       |
| ------------------- | ---------------------------------------------------------------------------- |
| Release narratives  | `plans/releases/` layout, `release.template.md`, scaffolded on `aps init`    |
| Release lint        | R001–R004 validate release files; `docs/release-planning.md` guide           |
| Solution library    | `solution.template.md` + per-project solutions archive (learn phase)         |

**APS modules:** [release-planning](../modules/release-planning.aps.md)

### Binary-first distribution (INSTALL)

APS stops vendoring a script and ships as a global binary across channels.

| Area                | Detail                                                                       |
| ------------------- | ---------------------------------------------------------------------------- |
| Install channels    | crates.io (`cargo install aps-cli`), GitHub release binaries, Scoop, `cargo binstall` |
| Minimal init        | `aps init` scaffolds project content only; no default local CLI vendoring    |
| Project contract    | `.aps/config.yml` pins `cli_version`; `plans_dir`/`docs_dir` discovery       |
| Migration           | Path from vendored CLI to the global binary; `aps setup` picker + shortcuts  |

**APS module:** [install](../modules/install.aps.md) (INSTALL-010…018 Complete
Complete; INSTALL-015 smoke passed on tag `v0.4.0`)

## Risks

| Risk                                                       | Impact | Mitigation                                                                 |
| ---------------------------------------------------------- | ------ | -------------------------------------------------------------------------- |
| INSTALL-015 smoke test only runs against a real tag        | Medium | Treat the v0.4.0 tag as the smoke test; verify each channel before announcing; hold the announcement until all four install paths resolve `0.4.0` |
| Version skew across channels (`0.4.0-dev` left behind)     | Medium | Bump `cli/Cargo.toml` **and** `.aps/config.yml` `cli_version` to `0.4.0` in the cut; one semver across channels (D-036). Clears the standing lint warning |
| crates.io publish is irreversible (cannot unpublish)       | Medium | Dry-run `cargo publish --dry-run`; verify the packaged `include` (symlinked scaffold/templates) before the real publish |
| Large surface (8 modules) outpaces the CHANGELOG narrative | Low    | This plan is the map; CHANGELOG `[0.4.0]` links here for the full story    |

## Out of Scope

- **examples / prompts / integrations / monorepo** — all Draft; deferred to v0.5+. Not in this cut.
- **ecosystem** (GitHub Action, VS Code extension) — Proposed / Long Term.
- **`aps release` subcommand** (REL-005) — deferred pending more release-planning mileage; release docs stay template-driven for v0.4.

## Rollout

- **Version bump:** `0.4.0-dev` → `0.4.0` in `cli/Cargo.toml` and `.aps/config.yml`.
- **Distribution:** tag `v0.4.0`; publish to crates.io, attach GitHub release binaries, update Scoop manifest, confirm `cargo binstall`.
- **Migration:** existing projects move from the vendored CLI to the global binary (INSTALL-017 path); `.aps/config.yml` pins `cli_version: 0.4.0`.
- **Communication:** promote CHANGELOG `[Unreleased]` → `[0.4.0]` linking this plan; README install section already points at the binary channels.
