## Release Theme

**Orchestration & Multi-Agent Reach** — APS grows from a markdown convention
into a workflow tool. `aps next/start/complete` give projects a CLI-driven
state machine for the Ready → In Progress → Complete lifecycle, with context
packages and learning capture wired into every transition. APS agents now
target five harnesses (Claude Code, Codex, Copilot, OpenCode, Gemini), and a
new `.aps/` consolidated layout keeps everything in one place per project.

## What Ships

### Orchestration CLI (ORCH)

The defining capability of this release. APS is no longer just a spec —
there's a tool that drives the lifecycle.

| Area              | Detail                                                                                |
| ----------------- | ------------------------------------------------------------------------------------- |
| Dependency walk   | `aps next` resolves the next ready work item across modules, honouring module + item deps |
| State machine     | `aps start <ID>` and `aps complete <ID>` enforce Ready → In Progress → Complete       |
| Context packaging | `aps start` writes `.aps/context/<ID>.md` with item, scope, decisions, deps           |
| Dependency graph  | `aps graph [module]` renders work items with upstream deps and per-item status        |
| Learning capture  | `aps complete --learning "..."` inserts a `- **Learning:**` line per ORCH D-002       |

**APS module:** [ORCH](../modules/orchestrate.aps.md) (ORCH-001 through ORCH-004 Complete)

### TUI Onboarding (TUI)

First Ratatui-based onboarding flow lands. Framework decision recorded in
[D-001 on the TUI module](../modules/tui.aps.md).

| Area              | Detail                                                       |
| ----------------- | ------------------------------------------------------------ |
| Init wizard       | TUI-001 ships the Ratatui-based `aps init` wizard            |
| Framework         | Rust + Ratatui, replacing earlier OpenTUI/Bun exploration    |

**APS module:** [TUI](../modules/tui.aps.md) (TUI-001 Complete, TUI-002 in progress)

### Layout v2 & Distribution (INSTALL, AGENTS)

| Area               | Detail                                                                       |
| ------------------ | ---------------------------------------------------------------------------- |
| Consolidated root  | `.aps/` houses tooling, context, and runtime files in one place              |
| Migration          | `aps migrate` converts v1 layouts to v2 with a shell-prompt wizard           |
| Global install     | `--global` flag for system-wide CLI installation                             |
| Multi-agent ports  | APS agents ported to Codex, GitHub Copilot, OpenCode, and Gemini             |
| Planner & Librarian| APS planner, librarian, and conductor agents added                           |
| Designs folder     | `designs/` added as a first-class APS artifact alongside specs and plans     |
| Wave execution     | Action plans support wave-based parallel execution guidance                  |

**APS modules:** [INSTALL](../modules/install.aps.md) (in progress follow-up),
[AGENTS](../modules/agents.aps.md) (Complete)

### Process & Branching

| Area              | Detail                                                                 |
| ----------------- | ---------------------------------------------------------------------- |
| Primary branch    | Canonical primary branch promoted from `dev` to `main`; CI updated     |
| Skill bootstrap   | `/plan` skill auto-bootstraps `aps init` and performs a version check  |
| Skill decoupled   | Skill install decoupled from `aps init` — install once globally        |

## Success Criteria

- [x] `aps next/start/complete` cycle works end-to-end on a fresh project
- [x] `aps complete --learning` surfaces in downstream `aps start` context
- [x] Multi-agent install works across all five supported harnesses
- [x] v1 → v2 migration is reversible (backups taken, dry-run available)
- [x] CHANGELOG and tag published; install script pulls v0.3.0 by default

## Risks

| Risk                                              | Impact | Mitigation                                                                  |
| ------------------------------------------------- | ------ | --------------------------------------------------------------------------- |
| v1 → v2 migration breaks existing projects        | High   | `aps migrate --dry-run`, automatic backups, rollback documented             |
| State-machine semantics confuse hand-edit users   | Medium | Hand-editing remains supported; CLI is opt-in; docs spell out both modes    |
| Multi-agent surface area outpaces test coverage   | Medium | Each agent install path exercised in CI; smoke test on common harnesses     |

## Out of Scope

- **Conductor module promotion** — `conductor.aps.md` remains a trial; not
  promoted to a first-class module type in this release.
- **`aps release` subcommand** — release narratives are doc-only for now (see
  [COMPOUND-003](../modules/compound.aps.md)). CLI helper deferred.
- **`aps archive <MODULE>` helper** — deferred per
  [COMPOUND D-001](../modules/compound.aps.md). Manual archival is fine for v0.3.

## Rollout

- **Distribution:** GitHub release tag `v0.3.0`; install script pulls from `main`
  by default and supports `APS_VERSION=v0.3.0` pinning.
- **Migration:** existing projects run `aps migrate` (dry-run available) to move
  to the v2 layout.
- **Communication:** [CHANGELOG entry](../../CHANGELOG.md#030---2026-05-20) and
  README refresh; the README overhaul landed in the same cut.
