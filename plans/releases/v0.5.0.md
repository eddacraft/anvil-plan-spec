## Release Theme

**Federated Plans & Full-Lifecycle Tooling** — v0.4 made APS installable as a
real binary and introduced crosscutting coordination. v0.5 lets that workflow
scale beyond one plan tree. Independently owned packages can keep standalone
plans beside their code, link them into a federation, and still lint,
orchestrate, and roll up the whole system from one parent. The native binary
also closes its lifecycle gaps, project upgrades become explicit and
reversible, and three-implementation parity moves from convention to a CI
contract.

## What Ships

### Federated nested plans (MONO)

The headline capability is an opt-in planning tier for packages with independent
owners, backlogs, or release cadences. Tagged modules remain the default for
smaller monorepos.

| Area               | Detail |
| ------------------ | ------ |
| Plan layout        | Co-located standalone child plans linked by a parent `## Child Plans` section |
| Cross-tree refs    | Bare IDs locally; `<child>:<ID>` references across plan boundaries |
| Federated lint     | Transitive traversal, prefix-aware W003, work-item W020, and module W021 collision warnings |
| Orchestration      | `next`, `start`, `complete`, `graph`, and `audit` traverse the tree and accept `--child` |
| Roll-up            | `aps rollup` renders child completion, next-ready work, and aggregate status |
| Scaffolding        | Bash `--scope nested` and Rust/TUI `--templates index-nested` create the same lint-clean tree |
| Guidance           | Tags-vs-nested decision table, migration steps, and a complete storefront example |

**APS module:** [monorepo](../modules/monorepo.aps.md) (MONO-001…006
Complete; MONO-007/008 merged for this cut)

### Full native lifecycle and safer project maintenance

The binary-first promise from v0.4 now covers the operational lifecycle, not
only onboarding and read-only plan inspection.

| Area                | Detail |
| ------------------- | ------ |
| Native lifecycle    | Rust implementations of `start`, `complete`, `graph`, and `audit`, parity-checked against bash |
| Footprint updates   | `aps update` adds missing generated files, refreshes owned files, and reports every result |
| Migration           | `aps migrate` diagnoses first, dry-runs by default, and backs up before removing vendored CLI files |
| Setup diagnostics   | `setup upgrade` explains absent templates; `doctor` points to the real migration command |

**APS modules:** [orchestrate](../modules/orchestrate.aps.md),
[install](../modules/install.aps.md)

### Cross-implementation parity (COND, CIP)

Rust, bash, and PowerShell are independent implementations of one APS command
contract. v0.5 gives that contract behavioral teeth.

| Area                 | Detail |
| -------------------- | ------ |
| Conductor lint       | W002/W006 restored in bash and PowerShell with Rust ordering and messages |
| PowerShell hardening | Status separator parsing and civil-day age calculations corrected |
| Behavioral CI        | Conductor/status fixtures execute under PowerShell rather than relying on string guards |
| Corpus diff          | CI compares ordered lint findings from all three CLIs across seven fixture trees |
| Policy               | D-039 requires maintained three-way lockstep for shared command surfaces |

**APS modules:** [conductor](../modules/conductor.aps.md) (COND-007 Complete),
[ci-parity](../modules/ci-parity.aps.md) (CIP-001/002 Complete)

### Canonical APS development loop

The repository now carries the canonical Claude and Codex skill set for taking
approved APS work from discovery through safe unattended execution.

| Area       | Detail |
| ---------- | ------ |
| Routing    | Planning and development workflow routers choose the appropriate lifecycle stage |
| Autonomy   | Gate policy and task routing tune ceremony without bypassing authority boundaries |
| Safety     | Immutable safety rails and product-vs-technical triage control unattended decisions |
| Resilience | Probe, durable resume, escalation queue, and crash-safe landing support long runs |

This is a repository delivery surface rather than a new APS format module; its
source is the installed `.claude/skills/` and `.codex/skills/` trees.

## Success Criteria

- [x] A parent and each standalone child lint clean; bad cross-tree references
  and duplicate IDs produce deterministic findings in Rust, bash, and
  PowerShell.
- [x] `next`, `start`, `complete`, `graph`, `audit`, and `rollup` operate on the
  nested-plan fixture without mutating the wrong child.
- [x] Native `start`/`complete` mutations, context packages, graphs, and audit
  output match the bash reference.
- [x] The full CLI suite, Cargo tests, clippy, rustfmt, APS lint, markdownlint,
  and `cargo publish --dry-run` pass for the release candidate.
- [ ] GitHub assets, crates.io, Scoop, and `cargo binstall` all resolve
  **v0.5.0**, and `aps --version` reports `0.5.0`.
- [ ] CHANGELOG `[Unreleased]` is promoted to `[0.5.0]`, release notes are
  ready, and the public tag is published.

Preparation evidence (2026-07-11): shell suite green; 146 Cargo tests; APS lint
clean across 34 files; clippy, rustfmt, and markdownlint clean; nested
orchestration green; bash/Rust findings identical across seven fixtures;
`cargo publish --dry-run --allow-dirty` packaged and verified 91 files. `pwsh`
is unavailable locally; the unchanged PowerShell and cross-CLI behavior is
covered by successful `main` CI at `ffd528f`.

## Risks

| Risk | Impact | Mitigation |
| ---- | ------ | ---------- |
| Cross-tree ambiguity mutates the wrong plan | High | Refuse ambiguous bare IDs, require child qualification, and test mutation isolation |
| Independent CLIs drift after the release | High | Keep D-039 mandatory and run the cross-CLI corpus diff in CI |
| crates.io publication cannot be undone | Medium | Run `cargo publish --dry-run` from the exact release commit before tagging |
| Scoop requires hashes that do not exist before the tag | Medium | Leave the manifest on v0.4.0 during preparation; update only from published `SHA256SUMS` |
| Existing monorepos assume one central backlog | Low | Nested plans are opt-in; tagged modules remain documented as the default tier |

## Out of Scope

- **Prompt variants** (PROMPTS-001…003) remain Ready and are not part of this
  cut.
- **`aps release` subcommand** (REL-005) remains deferred; release narratives
  stay template-driven.
- **External integrations and ecosystem wrappers** remain Draft / Long Term.
- **Post-release status closeout** for merged MONO items waits for tag and
  distribution evidence.

## Rollout

- **Version bump:** `0.4.0` → `0.5.0` in the Rust crate, project contract,
  private package metadata, installer defaults, scaffold stamps, and current
  examples.
- **Distribution:** tag `v0.5.0`; let the GitHub workflow publish five binary
  targets; publish `aps-cli` to crates.io; update Scoop from the release
  `SHA256SUMS`; confirm `cargo binstall`.
- **Migration:** no APS document migration is required. Existing tagged-module
  plans remain valid; federated plans are an opt-in layout.
- **Communication:** promote the CHANGELOG entry, update the README release
  summary and badge, then apply `docs/release-notes/v0.5.0.md` to the GitHub
  release created by the tag workflow.
