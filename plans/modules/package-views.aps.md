# Package Views (Tagged Monorepo Tier)

| ID  | Owner  | Priority | Status |
| --- | ------ | -------- | ------ |
| PKG | @aneki | high     | Ready  |

**Last reviewed:** 2026-07-13

## Purpose

Give the **tagged** monorepo tier (docs/monorepo.md) the tooling it never
received. The tier was created from anvil-001's pain — one canonical index
whose items regularly span packages — but shipped as docs and a template
column only: no linter reads `Packages:`, no command filters by it, and the
promised "What's Next" / "Modules by Package" views are hand-maintained
markdown that drifts. Meanwhile the rarer *nested* tier got full v0.5.0
tooling (traversal, W020/W021, orchestration, rollup, scaffold). This module
closes that inversion: the default tier gets first-class CLI support.

## In Scope

- `Packages:` awareness in `aps next` (filter and group-by).
- Lint validation of `Packages:` values against the workspace.
- Generated by-package views so the index sections stop drifting.

## Out of Scope

- Nested/federated plans (shipped — see [monorepo](./monorepo.aps.md)).
- Ownership semantics: `Packages:` stays a scope tag, not an owner field
  (docs/monorepo.md: the grouping "is a *view*, not ownership").

## Work Items

### PKG-001: `aps next --package <name>`

- **Intent:** Answer "what's next in the package I'm standing in" without
  reading the whole index
- **Expected Outcome:** `aps next --package <name>` restricts resolution to
  work items whose effective `Packages:` (item field, else module metadata)
  includes `<name>`; `aps next --by-package` groups the ready queue under
  package headings. Items with no `Packages:` appear in an "(untagged)"
  bucket so partial adoption stays visible rather than silently filtered.
- **Validation:** Fixture with tagged + untagged modules resolves the same
  filtered/next item in Rust, bash, and PowerShell; untagged bucket asserted
- **Confidence:** medium
- **Dependencies:** ORCH-001 (parser reuse)
- **Status:** Ready
- **Notes:** D-039 three-way parity applies (orchestration surface).

### PKG-002: Lint `Packages:` values (typo guard)

- **Intent:** A `Packages: adaptors` typo should warn, the same way W002/W006
  catch conductor reference typos
- **Expected Outcome:** New W-code: a `Packages:` entry (module metadata or
  work-item field) that resolves to no directory in the workspace warns with
  the offending value and line. Resolution checks the path as given plus
  conventional roots (`packages/<name>`, `apps/<name>`); silent when the
  project has no monorepo markers, so single-package projects never pay for
  the check.
- **Validation:** Fixtures: valid tags silent, typo'd tag warns, non-monorepo
  project silent — byte-identical across all three linters via
  `test/cli-parity.sh`
- **Confidence:** medium
- **Dependencies:** none
- **Status:** Ready
- **Notes:** D-039 parity; wire fixtures into the cross-CLI corpus so the
  CIP-002 harness covers the new code from day one.

### PKG-003: Generated by-package view

- **Intent:** Stop hand-maintaining the "Modules by Package" index section —
  generated views cannot drift
- **Expected Outcome:** `aps rollup --by-package` renders modules/items
  grouped by package from live metadata (the tagged-tier analogue of the
  nested tier's `aps rollup`). Optionally `--write` refreshes a marked
  index section between `<!-- aps:by-package -->` fences, leaving the rest
  of the index untouched.
- **Validation:** Rendered view matches module metadata on a tagged fixture;
  `--write` is idempotent and touches only the fenced block; parity suite
  green
- **Confidence:** medium
- **Dependencies:** PKG-001 (shared package-resolution helpers)
- **Status:** Ready
- **Notes:** Print-only is the MVP; `--write` can follow. D-039 parity.
