# TUI-003 Implementation Plan

**Goal:** Add Templates, Paths, and Components wizard sections to `aps init` so users control what gets installed and where.
**Architecture:** Extend the testable `WizardState` machine with three new steps inserted into the existing flow (Profile → Shape → **Templates** → AI Tooling → Tool Config → **Paths** → **Components** → Summary). Paths and the custom-template row use `eddacraft_tui::TextInputState`; because `KeyHandler::map` aliases `j/k/h/l/q/space` to navigation actions, a new `map_key(event, editing)` function bypasses the vim aliases while a text field is focused. The directory preview is a pure function of wizard state, unit-testable without a terminal.
**Tech Stack:** Rust 2024, clap, ratatui/crossterm, eddacraft-tui (Select, TextInput widgets).

## Design Notes

- **Step order** follows the module's In Scope listing: Templates after Project
  Shape; Paths after per-tool config; Components after Paths.
- **Template defaults** (recomputed on entering Templates until the user first
  toggles a row, tracked by `templates_touched`):
  - monorepo → `module` + `monorepo-index`
  - single + solo → `quickstart`
  - single + team/agent-operator → `index` + `module`
- **Edit mode:** `WizardState::is_editing()` is true on the Paths step and on
  the Templates step while the Custom row's input is active. `map_key` passes
  `Char(c)` through raw in edit mode (so `j`, `q`, space are typable),
  keeps Ctrl+C → Quit, and maps arrows/Enter/Esc/Backspace/Delete/Home/End
  to their actions. Arrow keys (not `j`/`k`) switch fields on the Paths step.
- **Path defaults:** plans dir `plans/`, docs location `docs/`, tooling root
  `.aps/`.
- **Components** (all default on): lint rules, aps-rules.md,
  project-context.md, designs/ directory, decisions/ directory.
- **Scaffold-output validation** ("custom paths produce valid scaffold")
  executes in TUI-004 — this item validates that wizard state captures the
  selections correctly, mirroring how TUI-001 deferred the widget harness.
