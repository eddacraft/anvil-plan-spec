# APS v0.7.0 — Team Foundations

v0.6 finished the monorepo tiers. v0.7 makes APS adoptable by a *team*, not
just its author — it targets everything a new team hits in week one: managed
installs that survive updates, one layout across every CLI, a machine-readable
status surface, central lint enforcement in CI, and written conventions.

## Highlights

- **Managed skill installs** — every installed skill tree now carries an
  `.aps-managed.json` marker (SHA-256 inventory + `cliVersion`). `aps update`
  reconciles against it: fresh trees stay put, stale ones refresh, and trees
  you've edited are refused, never clobbered. The marker is written and
  reconciled **byte-for-byte identically by the Rust, bash, and PowerShell
  CLIs** — a tree installed by one can be verified and updated by any other —
  and `aps doctor` reports each root as fresh/stale/dirty/unmanaged/broken.
- **The v2 layout, everywhere** — the `curl | bash` and `irm | iex` updaters
  and the PowerShell `aps init` no longer install (or leave behind) the old
  layout. They deliver the current `.aps/`-based layout, migrating v1 projects
  with a backup of anything they remove. `cli_version` in `.aps/config.yml` is
  now the single on-disk version stamp — the retired `plans/.aps-version` is
  removed on update.
- **`aps export --json`** — a compact, deterministic `aps-export/v1` snapshot
  of the whole plan tree (modules, work items, statuses, dependency tokens,
  package tags) for dashboards and non-CLI stakeholders. Byte-identical in the
  Rust and bash CLIs.
- **Composite GitHub Action** — `uses: eddacraft/anvil-plan-spec@<tag>` lints
  a repo's plans in CI with the action ref pinning the CLI version; an opt-in
  `rollup-comment` keeps a sticky PR comment with the federated rollup or the
  ready queue. Dogfooded by this repo's own CI.
- **Harness set revision (Gemini out, Grok in)** — the supported set is now
  Claude Code, Copilot, Codex, OpenCode, and Grok Build. Grok rides the
  Codex-shared `.agents/skills/` payload with no bespoke assets; `gemini` as a
  tool value errors with a pointer, and existing installs are untouched.
- **Fence-aware parsing** — work-item headers and fields inside fenced code
  blocks are treated as documentation examples in all three CLIs: no more
  false `E005`s or phantom `next`/`graph` entries from example snippets.
- **Team rollout guide + multi-owner example** — `docs/team-rollout.md`
  (ownership, plan-change review, merge-conflict conventions, version pinning,
  the Windows/PowerShell story) and `examples/team-payments/`, a lint-clean
  three-owner plan mid-execution.

## Upgrading

Existing projects: `aps update` migrates you to the current layout and
reconciles skill trees safely (your edits are preserved, not overwritten). The
CLI itself upgrades the way you installed it:

```bash
cargo install aps-cli            # or: cargo binstall aps-cli
curl -fsSL https://raw.githubusercontent.com/eddacraft/anvil-plan-spec/main/scaffold/install | bash -s -- --cli
# Scoop (Windows)
scoop install https://raw.githubusercontent.com/eddacraft/anvil-plan-spec/main/packaging/scoop/aps.json
```

See [docs/team-rollout.md](https://github.com/eddacraft/anvil-plan-spec/blob/main/docs/team-rollout.md)
